1. Tech Stack & Architecture

InterroPedago is designed to be highly portable and serverless-friendly, operating entirely on flat files without requiring a SQL/NoSQL engine.

  • Backend Framework: FastAPI (Python 3.10+) running on Uvicorn.
  • Frontend: Vanilla HTML/CSS/JS. No bundlers (Webpack/Vite) required. Served statically by FastAPI via StaticFiles.
  • Database: Custom JSON-based ORM (backend/database.py). File I/O operations act as the transactional layer.
  • AI Provider: Google Gemini API (google-generativeai). Specifically using gemini-3.1-pro-preview for multi-modal context parsing.
  • LaTeX Compilation: Shell calls to pdflatex. Requires a working TeX Live distribution on the host server.

2. Data Schema (Flat Files)

The data/ directory is mounted as a static directory for serving generated PDFs and parsed scans.

data/
├── global/
│   ├── students.json     # Registry of all students [{id, name, password, ...}]
│   ├── courses.json      # Registry of all courses [{id, title, enrolled_students, ...}]
│   └── quizzes.json      # Global index of generated quizzes
├── courses/
│   └── {course_id}/
│       ├── {course_name}.tex            # Original source file
│       ├── question_bank.json           # Validated LLM-generated questions
│       ├── quiz_{n}_selected_bank.json  # Snapshot of questions used for quiz N
│       ├── quiz_{n}_summary.md          # Generated global synthesis
│       └── attendance.json              # Course attendance matrix
└── students/
    └── {student_id}/
        ├── history.json                 # Fast-access history for the Chatbot
        └── {course_id}/
            └── quiz_{n}/
                ├── subject.pdf          # Generated unique subject
                ├── scan.pdf             # Uploaded student copy
                └── correction.json      # Smart Grading JSON output (grades & feedback)

3. API Reference: Students

GET /api/students

Returns the list of all students augmented with calculated stats (absences, average_grade, course_stats) computed on the fly from the gradebooks.

POST /api/students/import_csv

Accepts a multipart/form-data upload containing a CSV. Parses delimiter automatically (;, , \t). Deduplicates homonyms by appending suffixes like (2).

4. API Reference: Courses

POST /api/courses/{course_id}/settings

Updates the course configuration including the active LaTeX bounds (start_line, end_line) and specific prompts (generation, grading, summary).

GET /api/courses/{course_id}/preview

Returns a preview of the LaTeX source code based on the current bounds, injecting line numbers dynamically for the frontend to render.

5. API Reference: Generation (LLM)

POST /api/courses/{course_id}/generate NDJSON Stream

Triggers the personalized quiz generation pipeline for all enrolled students.

Payload: {"n_new": 2, "n_old": 2, "selected_questions": "[...]", "regenerate": false}

Response: Streaming application/x-ndjson. Yields progress events, followed by a final result event with the global compiled PDF URL.

// Example Stream Output:
{"type": "progress", "current": 1, "total": 20, "message": "Generating for 1234..."}
...
{"type": "result", "count": 20, "global_pdf_url": "/content/courses/math/all_quizzes_3.pdf"}
POST /api/courses/{course_id}/generate_bank

Synchronous endpoint. Sends the course segment and bank prompt to Gemini to extract num_questions formatted as LaTeX \item blocks.

6. API Reference: Grading (OCR/LLM)

POST /api/grading/smart NDJSON Stream (Bulk)

Handles the core "Smart Grading" logic.

If is_bulk=true, the backend uses pypdf to chunk the uploaded PDF into blocks of pages_per_copy.

Flow:

  1. OCR Identification (Gemini): Extracts Student ID and Quiz Number.
  2. Database Lookup: Retrieves the specific PDF subject generated for this student.
  3. Multi-modal Grading (Gemini): The model is fed both PDFs (Subject + Scan) and asked to extract transcription and provide feedback.

To avoid event loop blocking during long LLM calls, the grading process is wrapped in fastapi.concurrency.run_in_threadpool.

7. Streaming Patterns (NDJSON)

Due to the high latency of LLM calls, standard HTTP responses lead to frontend timeouts. V6 implements Newline Delimited JSON (NDJSON) via StreamingResponse.

The frontend parses the stream using the Fetch API and TextDecoderStream:

const response = await fetch(url, { ... });
const reader = response.body.pipeThrough(new TextDecoderStream()).getReader();

while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    const lines = value.split('\n').filter(l => l.trim() !== '');
    for (const line of lines) {
        const data = JSON.parse(line);
        // Handle data.type == "progress" or "result"
    }
}

8. Deployment & Sync

The backend includes an /api/sync endpoint (used by the Déployer dropdown) which accepts a ZIP payload and a JSON list of deletions.

It acts as a rudimentary CI/CD pipeline, unzipping the payload directly into the active data/ directory (or codebase directory) to update the remote university server without requiring SSH access.

9. Local Setup & Installation

To run the InterroPedago V6 environment locally for development:

  1. Python Version: Ensure Python 3.10+ is installed.
  2. System Dependencies: Install texlive-full (or equivalent) for LaTeX compilation. 
    sudo apt-get install texlive-full
  3. API Key: Create a file named Gemini_key in the root of the teacher and student directories, containing only your Google Gemini API key. Alternatively, export it as an environment variable GEMINI_API_KEY.
  4. Dependencies: Install the Python requirements. 
    pip install fastapi uvicorn google-generativeai pypdf python-multipart json-repair
  5. Running the Servers: Execute the startup scripts from their respective directories. 
    cd teacher && ./run_server.sh
# In another terminal:
cd student && ./run_server.sh

10. Python Modules Breakdown

The backend logic is split into several focused modules located in backend/:

  • main.py: The FastAPI application entry point. Defines all API routes, mounts static directories, and orchestrates calls to other modules.
  • database.py: A custom JSON ORM. Handles thread-safe (or semi-safe) reads/writes to the data/ directory files (students, courses, history). Contains functions like get_all_students(), save_course(), etc.
  • generator.py: Handles the LLM integration for quiz and bank generation. It constructs the prompts, calls the Gemini API, and uses latex_utils.py to compile the resulting LaTeX code into PDFs.
  • grader.py: Implements the Smart Grading logic. Orchestrates the OCR phase (identifying the student from the scan) and the grading phase (sending the scan and subject to Gemini for evaluation).
  • gradebook.py: Manages the extraction and aggregation of grades from the decentralized correction.json files into matrix formats suitable for the frontend display.
  • chatbot.py: Contains the logic for the student-facing AI tutor. It injects the student's specific history into the system prompt to provide contextual pedagogical help.
  • latex_utils.py: Utility functions for interacting with the system's pdflatex command, handling temporary directories, escaping strings, and managing compilation errors.

11. Prompt Engineering Specifics

The system's intelligence relies heavily on specific prompts. Here are the core architectures of the default prompts:

Quiz Generation Prompt (generator.py)

Variables Injected: {history}, {context}, {selected_questions}, {n_new}, {n_old}, {n_total}.

Key Directives:

  • "RETURN ONLY THE LATEX CODE inside the enumerate."
  • "CRITICAL: NO ANSWER BOXES, NO FRAMES... Do not leave space or lines for the student to write their answer."
  • "ENSURE EACH QUESTION IS IN ENGLISH AND FRENCH."
Smart Grading Prompt (grader.py)

Context Files: The generated Subject PDF (File 1) and the Student's Scan (File 2).

Key Directives:

  • "Transcribe the student's answer for each question."
  • "Assign a score /20 (or as specified)."
  • "Output JSON with strict schema: grades [{question_number, transcription, score, feedback}]."

Note: The backend uses the json-repair library to handle occasional hallucinated characters outside the JSON block.

12. Complex Frontend Logic

Dynamic Resizers

The LaTeX viewer and Prompt Editors feature draggable resize bars. This is implemented via native Javascript mouse events (mousedown, mousemove, mouseup) attached to the #courseDocumentResizer and .resizer-bar elements. The mousemove event dynamically adjusts the height or flex-basis of the parent container.

LaTeX Rendering

When the user clicks "👁️ Prévisualiser", the frontend fetches the raw text from the backend's /preview endpoint. It does not compile it to PDF. Instead, it renders the raw text in a <pre> tag and relies on MathJax (loaded via CDN in the <head>) to asynchronously parse and render any inline ($...$) or block ($$...$$) equations.

State Management

Since the application is vanilla JS, state is often maintained in DOM elements (e.g., hidden inputs like #courseSelector) or simple global variables. The loadGradebook() and loadAttendance() functions completely rebuild their respective HTML tables upon successful fetch requests, ensuring the UI remains synchronized with the backend data.

13. Architecture: Student Server & Chatbot

The student application (student_server) runs on a separate Uvicorn instance (port 8001) to isolate student access from the teacher dashboard.

Authentication

Students log in using their ID and the generated password. Authentication is session-based (via cookies) or handled locally in the frontend depending on the setup. The server verifies credentials against data/global/students.json.

The AI Chatbot Logic

The chatbot (student/backend/chatbot.py) is not a generic LLM. It is heavily contextualized.

  1. When a student opens the chat, the system loads their history.json, which contains a record of all past quizzes, the student's specific answers, the grades, and the teacher/AI feedback.
  2. This history is injected into the Gemini System Prompt.
  3. If a student asks "Why did I fail question 2 last week?", the AI has direct access to the exact text they wrote and the correction applied, allowing it to act as a highly personalized tutor.

15. Troubleshooting Guide & Error Handling

1. LaTeX Compilation Errors

Symptom: Generation stops midway, or PDF is not created.

Cause: The AI generated invalid LaTeX syntax (e.g., unescaped characters like # or & inside equations).

Fix: Check teacher/teacher_server.log for the raw pdflatex output. Adjust the "Prompt de Génération" to explicitly instruct Gemini to escape specific characters, or modify latex_utils.py to run a regex cleaning function before compilation.

2. Gemini API Rate Limits (429 Too Many Requests)

Symptom: Stream fails with "Quota Exceeded" or "Resource Exhausted".

Fix: The Smart Grading and Generation loops might be hitting the API too fast. Implement asyncio.sleep(2) between iterations in main.py loops, or upgrade the Google Cloud project to a paid tier to increase RPM (Requests Per Minute).

3. JSON Parsing Failures from LLM

Symptom: json.decoder.JSONDecodeError during grading.

Fix: Gemini sometimes outputs markdown blocks (```json ... ```) or prepends conversational text. The backend already uses json-repair to mitigate this. If it still fails, reinforce the prompt: "Return ONLY the raw JSON. Absolutely no other text."

16. Extending the System

How to add a new feature (e.g., a "Delete All Quizzes" button):

  1. Backend API (main.py): Add the route. 
    @app.delete("/api/courses/{course_id}/quizzes")
async def delete_all_quizzes(course_id: str):
    database.delete_course_quizzes(course_id)
    return {"status": "success"}
  2. Database Layer (database.py): Implement the logic using pathlib. 
    def delete_course_quizzes(course_id):
    course_dir = COURSES_DIR / course_id
    # Logic to delete quiz_* folders via shutil.rmtree
    return True
  3. Frontend Interface (index.html): Add the button. 
    <button onclick="deleteAllQuizzes()">Delete All</button>
  4. Frontend Logic (index.html script): Implement the fetch call. 
    async function deleteAllQuizzes() {
    await fetch(`/api/courses/${currentCourseId}/quizzes`, { method: 'DELETE' });
    location.reload(); // Refresh the UI
}
